# 커피 주문 앱

## 1. 프로젝트 개요

### 1.1 프로젝트명
커피 주문 앱

### 1.2 프로젝트 목적
사용자가 커피 메뉴를 주문하고, 관리자가 주문을 관리할 수 있는 간단한 풀스택 웹 앱

### 1.3 개발 범위
- 주문하기 화면(메뉴 선택 및 장바구니 기능)
- 관리자 화면(재고 관리 및 주문 상태 관리)
- 데이터를 생성/조회/수정/삭제할 수 있는 기능

## 2. 기술 스택
- 프런트엔드: HTML, CSS, 리액트, 자바스크립트
- 백엔드: Node.js, Express
- 데이터베이스: PostgreSQL

## 3. 기본 사항
- 프런트엔드와 백엔드를 따로 개발
- 기본적인 웹 기술만 사용
- 학습 목적이므로 사용자 인증이나 결제 기능은 제외
- 메뉴는 커피 메뉴만 있음

## 4. 주문하기 화면 PRD

### 4.1 문서 목적
`주문하기` 화면의 UI/UX 요구사항과 동작 기준을 정의하여,
프런트엔드 구현과 QA 기준을 일관되게 맞춘다.

### 4.2 대상 사용자
- 매장에서 음료를 직접 선택해 주문하는 일반 사용자

### 4.3 화면 목표
- 사용자가 메뉴를 빠르게 탐색하고 옵션을 선택해 장바구니에 담을 수 있어야 한다.
- 장바구니에서 주문 내역과 총 금액을 즉시 확인하고 주문을 완료할 수 있어야 한다.

### 4.4 화면 범위
- 상단 네비게이션(로고, `주문하기`, `관리자`)
- 메뉴 카드 목록 영역
- 장바구니 요약 영역
- 주문 실행 버튼

### 4.5 정보 구조(IA)

#### A. 상단 네비게이션
- 좌측: 브랜드 텍스트(`COZY`)
- 우측: 탭 2개(`주문하기`, `관리자`)
- 현재 화면인 `주문하기`는 활성 상태로 표시

#### B. 메뉴 카드 목록
각 카드에 다음 정보와 액션을 표시한다.
- 메뉴 이미지(없을 경우 기본 플레이스홀더)
- 메뉴명(예: 아메리카노(ICE), 아메리카노(HOT), 카페라떼)
- 기본 가격(원화 표기)
- 한 줄 설명
- 옵션 체크박스 목록
  - 샷 추가(+500원)
  - 시럽 추가(+0원)
- `담기` 버튼

#### C. 장바구니 영역
- 섹션 타이틀: `장바구니`
- 항목별 정보
  - 메뉴명 + 선택 옵션
  - 수량(`x n`)
  - 항목 합계 금액
- 우측 하단
  - `총 금액`
  - `주문하기` 버튼

### 4.6 사용자 시나리오
1) 사용자는 메뉴 카드를 보고 원하는 메뉴를 선택한다.  
2) 옵션(샷/시럽)을 체크한다.  
3) `담기` 버튼을 누르면 장바구니에 항목이 추가된다.  
4) 동일한 구성(메뉴+옵션)을 다시 담으면 수량이 증가한다.  
5) 장바구니 총 금액이 실시간으로 갱신된다.  
6) `주문하기` 버튼 클릭 시 주문이 생성된다.

### 4.7 기능 요구사항

#### FR-01 메뉴 목록 조회/표시
- 시스템은 주문 가능한 커피 메뉴 목록을 카드 형태로 표시해야 한다.
- 메뉴별 이미지, 이름, 가격, 설명을 노출해야 한다.

#### FR-02 옵션 선택
- 시스템은 메뉴별 옵션 체크박스를 제공해야 한다.
- 옵션 선택에 따라 항목 가격이 변경되어야 한다.

#### FR-03 장바구니 담기
- 사용자가 `담기`를 클릭하면 선택한 메뉴/옵션 조합이 장바구니에 추가되어야 한다.
- 동일 조합이 이미 있으면 새 줄을 만들지 않고 수량을 증가시켜야 한다.

#### FR-04 장바구니 요약 표시
- 장바구니는 항목명, 옵션, 수량, 항목 합계를 표시해야 한다.
- 총 금액은 모든 항목 합계의 합으로 계산되어야 한다.

#### FR-05 주문 요청
- `주문하기` 버튼 클릭 시 현재 장바구니 기준으로 주문 생성 요청을 전송해야 한다.
- 요청 성공 시 장바구니는 초기화되어야 한다.

#### FR-06 네비게이션
- `관리자` 탭 클릭 시 관리자 화면으로 이동해야 한다.

### 4.8 상태 및 예외 처리 요구사항

#### 상태
- 메뉴 로딩 중: 로딩 UI 표시
- 메뉴 로딩 실패: 오류 메시지 표시 + 재시도 수단 제공
- 장바구니 비어 있음: 빈 상태 메시지와 안내 문구 표시
- 주문 처리 중: `주문하기` 버튼 비활성화

#### 예외
- 메뉴 데이터가 비어 있으면 사용자에게 안내 메시지를 노출한다.
- 주문 실패 시 실패 원인을 표시하고 재시도 가능해야 한다.
- 네트워크 오류 시 사용자에게 명확한 오류 메시지를 제공한다.

### 4.9 UI/표현 요구사항
- 금액은 `#,###원` 형식으로 표기한다. (예: 12,500원)
- 카드와 장바구니는 시각적으로 분리된 컨테이너로 제공한다.
- CTA 버튼(`담기`, `주문하기`)은 클릭 가능한 상태를 명확히 보여야 한다.
- 화면 너비가 줄어들면 카드 목록은 줄바꿈되어 표시되어야 한다.

### 4.10 비기능 요구사항
- 첫 화면 진입 후 메뉴 목록은 2초 이내에 사용자에게 표시되어야 한다.
- 사용자 조작(옵션 체크/담기) 후 장바구니 UI 반영은 즉시 이루어져야 한다.
- 일반적인 데스크톱 해상도(최소 1280px)에서 레이아웃이 깨지지 않아야 한다.

### 4.11 수용 기준(AC)
- AC-01: 메뉴 카드에 이미지/이름/가격/설명/옵션/담기 버튼이 모두 보인다.
- AC-02: 옵션 선택 후 담기 시 옵션 가격이 반영된 금액으로 장바구니에 추가된다.
- AC-03: 동일 메뉴+동일 옵션을 2회 담으면 수량만 증가하고 총 금액이 누적된다.
- AC-04: 장바구니에 항목이 1개 이상 있을 때 총 금액이 정확히 계산된다.
- AC-05: 장바구니가 비어 있으면 `주문하기` 버튼은 비활성화된다.
- AC-06: 주문 성공 시 장바구니가 비워지고 성공 상태를 사용자에게 보여준다.
- AC-07: 메뉴 조회 실패/주문 실패 시 오류 메시지와 재시도 동선이 제공된다.

## 5. 관리자 화면 PRD

### 5.1 문서 목적
`관리자` 화면에서 주문 처리 현황과 재고를 실시간에 가깝게 관리할 수 있도록
UI/UX 요구사항과 동작 기준을 정의한다.

### 5.2 대상 사용자
- 매장 운영자(관리자)

### 5.3 화면 목표
- 현재 주문 처리 상태를 한눈에 확인할 수 있어야 한다.
- 메뉴별 재고 수량을 빠르게 증감할 수 있어야 한다.
- 주문 상태를 접수부터 완료까지 일관되게 변경할 수 있어야 한다.

### 5.4 화면 범위
- 상단 네비게이션(로고, `주문하기`, `관리자`)
- 관리자 대시보드(주문 집계)
- 재고 현황 카드 목록(수량 조정)
- 주문 현황 목록(주문 상태 관리)

### 5.5 정보 구조(IA)

#### A. 상단 네비게이션
- 좌측: 브랜드 텍스트(`COZY`)
- 우측: 탭 2개(`주문하기`, `관리자`)
- 현재 화면인 `관리자`는 활성 상태로 표시

#### B. 관리자 대시보드 영역
- 섹션 타이틀: `관리자 대시보드`
- 집계 항목
  - 총 주문 수
  - 주문 접수 수
  - 제조 중 수
  - 제조 완료 수

#### C. 재고 현황 영역
- 섹션 타이틀: `재고 현황`
- 메뉴별 카드 정보
  - 메뉴명(예: 아메리카노(ICE), 아메리카노(HOT), 카페라떼)
  - 현재 재고 수량(예: 10개)
  - 재고 감소 버튼(`-`)
  - 재고 증가 버튼(`+`)

#### D. 주문 현황 영역
- 섹션 타이틀: `주문 현황`
- 주문 항목 정보
  - 주문 시각
  - 주문 메뉴/수량
  - 결제 금액
  - 현재 주문 상태 버튼(예: `주문 접수`)

### 5.6 사용자 시나리오
1) 관리자는 `관리자` 화면에 진입해 대시보드 요약 수치를 확인한다.  
2) 재고가 부족한 메뉴를 찾아 `+`/`-` 버튼으로 수량을 조정한다.  
3) 신규 주문이 들어오면 주문 현황에서 상태를 확인한다.  
4) 주문 상태를 `주문 접수` → `제조 중` → `제조 완료`로 변경한다.  
5) 상태 변경 결과가 대시보드 집계와 주문 목록에 즉시 반영된다.

### 5.7 기능 요구사항

#### FR-01 관리자 데이터 조회
- 시스템은 화면 진입 시 대시보드 집계, 재고 목록, 주문 목록을 조회해야 한다.
- 조회 실패 시 각 영역에 오류 상태를 표시해야 한다.

#### FR-02 주문 집계 표시
- 시스템은 주문 상태별 건수를 계산해 대시보드에 표시해야 한다.
- 집계 항목은 총 주문, 주문 접수, 제조 중, 제조 완료를 포함해야 한다.

#### FR-03 재고 수량 조정
- 관리자가 `+` 버튼 클릭 시 해당 메뉴 재고를 1 증가시켜야 한다.
- 관리자가 `-` 버튼 클릭 시 해당 메뉴 재고를 1 감소시켜야 한다.
- 재고는 0 미만으로 내려갈 수 없어야 한다.
- 재고 변경 성공 시 화면 수량은 즉시 갱신되어야 한다.

#### FR-04 주문 목록 표시
- 시스템은 최신 주문 기준으로 주문 목록을 표시해야 한다.
- 각 주문 항목에는 주문 시각, 메뉴/수량, 금액, 현재 상태가 표시되어야 한다.

#### FR-05 주문 상태 변경
- 관리자는 주문 상태를 단계적으로 변경할 수 있어야 한다.
- 상태 전이 규칙은 `주문 접수` → `제조 중` → `제조 완료`를 따라야 한다.
- `제조 완료` 상태의 주문은 다음 상태로 변경할 수 없어야 한다.

#### FR-06 네비게이션
- `주문하기` 탭 클릭 시 주문하기 화면으로 이동해야 한다.

### 5.8 상태 및 예외 처리 요구사항

#### 상태
- 초기 로딩: 대시보드/재고/주문 영역에 로딩 UI 표시
- 영역별 로딩: 재고 변경 또는 상태 변경 중 해당 버튼 비활성화
- 데이터 없음:
  - 주문 없음: `현재 접수된 주문이 없습니다.` 메시지 표시
  - 재고 목록 없음: `등록된 메뉴 재고가 없습니다.` 메시지 표시

#### 예외
- 데이터 조회 실패 시 영역별 오류 메시지와 재시도 동선을 제공한다.
- 재고 업데이트 실패 시 수량을 이전 값으로 롤백하고 실패 메시지를 표시한다.
- 주문 상태 업데이트 실패 시 이전 상태를 유지하고 실패 메시지를 표시한다.
- 네트워크 오류 시 사용자가 이해 가능한 문구로 오류를 안내한다.

### 5.9 UI/표현 요구사항
- 금액은 `#,###원` 형식으로 표기한다.
- 재고는 `n개` 단위로 표시한다.
- 주문 상태는 버튼 또는 배지 형태로 현재 단계를 명확히 구분한다.
- 재고/주문 영역은 카드형 컨테이너로 시각적으로 분리한다.
- 데스크톱 너비 기준에서 재고 카드는 가로 정렬, 좁은 화면에서는 줄바꿈되어야 한다.

### 5.10 비기능 요구사항
- 화면 진입 후 핵심 데이터(대시보드/재고/주문)는 2초 이내 표시되어야 한다.
- 재고 변경/상태 변경 액션 후 UI는 1초 이내 갱신되어야 한다.
- 동일 액션의 중복 요청 방지를 위해 처리 중 버튼은 비활성화되어야 한다.

### 5.11 수용 기준(AC)
- AC-01: 관리자 진입 시 대시보드, 재고 현황, 주문 현황 3개 영역이 보인다.
- AC-02: 대시보드에 총 주문/주문 접수/제조 중/제조 완료 수치가 표시된다.
- AC-03: 재고 `+` 클릭 시 수량이 1 증가하고 `-` 클릭 시 1 감소한다.
- AC-04: 재고가 0일 때 `-` 클릭으로 음수 재고가 되지 않는다.
- AC-05: 주문 상태는 `주문 접수`에서 `제조 중`, `제조 완료` 순서로만 변경된다.
- AC-06: `제조 완료` 주문은 추가 상태 변경이 불가능하다.
- AC-07: 상태/재고 변경 결과가 목록과 대시보드 집계에 반영된다.
- AC-08: 조회/업데이트 실패 시 사용자에게 오류 메시지와 재시도 동선이 제공된다.
